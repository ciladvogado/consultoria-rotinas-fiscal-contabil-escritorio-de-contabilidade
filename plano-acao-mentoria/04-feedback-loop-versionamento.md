# Processo de Feedback Loop e Versionamento

**Versão:** 2.0  
**Data de Criação:** 06/03/2026

---

## Visão Geral

O **Feedback Loop** é o coração do sistema de aprendizado contínuo. Cada cliente alimenta o próximo através de:

1. **Extração de Insights** — o que funcionou, o que não funcionou
2. **Consolidação de Aprendizados** — padrões e tendências
3. **Melhoria do Plano** — atualizar documentos e processos
4. **Versionamento** — rastrear evolução (v2.0 → v2.1 → v2.2...)
5. **Documentação** — registrar mudanças no CHANGELOG

---

## Fluxo do Feedback Loop

```
Cliente 1 Completa Mentoria
    ↓
Você Consolida Aprendizados
    ↓
Você Identifica Melhorias
    ↓
Você Atualiza Plano (v2.0 → v2.1)
    ↓
Você Documenta Mudanças (CHANGELOG)
    ↓
Cliente 2 Inicia com Plano v2.1
    ↓
Cliente 2 Completa Mentoria
    ↓
Você Consolida Novos Aprendizados
    ↓
Você Atualiza Plano (v2.1 → v2.2)
    ↓
... e assim por diante
```

## Sistema de Versionamento

### Nomenclatura

O plano segue versionamento semântico: **MAJOR.MINOR.PATCH**

- **MAJOR (2.x.x):** Mudanças estruturais grandes (ex: nova fase, novo módulo)
- **MINOR (x.1.x):** Melhorias significativas (ex: novo protocolo, novo KPI)
- **PATCH (x.x.1):** Correções e ajustes pequenos (ex: typo, clarificação)

### Histórico de Versões

| Versão | Data | Mudanças Principais | Cliente |
|---|---|---|---|
| 1.0 | 06/03/2026 | Versão inicial do plano | N/A |

## Processo de Atualização de Versão

### Passo 1: Consolidar Feedback
1. Ao final da mentoria, crie `feedback-consolidado-cliente-NOME.md`
2. Preencha todos os campos com honestidade
3. Identifique padrões e insights

### Passo 2: Decidir Tipo de Versão
1. Revise o feedback consolidado
2. Determine: MAJOR / MINOR / PATCH?
3. Liste mudanças planejadas

### Passo 3: Atualizar Documentos
1. Abra cada documento (01, 02, 03, 04)
2. Aplique mudanças conforme feedback
3. Atualize versão no topo de cada arquivo

### Passo 4: Atualizar CHANGELOG
1. Adicione nova entrada com versão e data
2. Liste mudanças em categorias: Adicionado, Alterado, Removido, Corrigido
3. Inclua feedback que levou às mudanças

### Passo 5: Commit no GitHub
```bash
git add -A
git commit -m "v2.1: Melhorias após Cliente 1 — simplificar POPs, adicionar templates visuais"
git push origin master
```

### Passo 6: Documentar em Histórico
1. Atualize tabela de histórico de versões
2. Registre cliente que levou à mudança
3. Salve feedback consolidado em pasta de histórico

## Exemplo Prático

### Cliente 1 Completa (Semana 12)

**Feedback Consolidado Identifica:**
- Estrutura de POPs com 10 tópicos é muito extensa → reduzir para 8
- Faltou mais tempo prático → adicionar 1h de prática por semana
- Kanban com muitos filtros confundiu → simplificar filtros padrão

**Decisão:** MINOR version bump (v2.0 → v2.1)

**Mudanças Aplicadas:**
1. Editar `01-plano-acao-3-meses.md` — reduzir POPs de 10 para 8 tópicos
2. Editar `02-protocolo-reuniao.md` — adicionar 1h de prática por semana
3. Editar `03-controle-andamento.md` — simplificar filtros padrão do Kanban
4. Atualizar `04-feedback-loop-versionamento.md` — versão 2.1

**CHANGELOG:**
```markdown
## [2.1] - 15/05/2026

### Adicionado
- 1 hora de prática semanal no cronograma

### Alterado
- Reduzir tópicos de POPs de 10 para 8 (remover 2 tópicos menos críticos)
- Simplificar filtros padrão do Kanban (de 9 para 5 filtros)

### Removido
- Filtros avançados do Kanban (mover para documentação opcional)

### Corrigido
- Clarificar estrutura de delegação na Semana 6

### Feedback do Cliente
- Estrutura de POPs com 10 tópicos é muito extensa
- Faltou mais tempo prático durante as reuniões
- Kanban com muitos filtros confundiu equipe
```

**Commit:**
```bash

```

## Dicas Práticas

1. **Honestidade:** Registre o que não funcionou com a mesma ênfase do que funcionou.
2. **Rapidez:** Preencha feedback consolidado enquanto a mentoria está fresca (até 1 semana após conclusão).
3. **Rastreabilidade:** Sempre vincule feedback a versões específicas no CHANGELOG.
4. **Compartilhamento:** Compartilhe CHANGELOG com clientes para transparência.
5. **Iteração:** Não espere 3 meses para melhorar. Faça ajustes pequenos durante a mentoria (PATCH versions).
